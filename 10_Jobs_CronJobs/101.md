Ejemplos de Job

La regla de oro para cualquier Job o CronJob es que la política de reinicio (restartPolicy) del Pod nunca puede ser Always (que es el valor por defecto en Kubernetes). Debe ser Never o OnFailure.
1. Job Básico (Hola Mundo)

Este es el Job más sencillo. Levanta un contenedor ligero (Alpine Linux), imprime un mensaje en la consola y termina.
YAML
```
apiVersion: batch/v1
kind: Job
metadata:
  name: hola-mundo-job
spec:
  template:
    spec:
      containers:
      - name: hola
        image: alpine
        command: ["echo", "¡Hola, Kubernetes! Tarea completada."]
      restartPolicy: Never
```
    Cuándo usarlo: Scripts de migración de bases de datos de una sola vez o tareas de inicialización.

2. Job con Paralelismo

Este Job debe ejecutarse con éxito un total de 3 veces (completions: 3), pero Kubernetes ejecutará hasta 2 de ellos al mismo tiempo (parallelism: 2) para terminar más rápido.
YAML
```
apiVersion: batch/v1
kind: Job
metadata:
  name: trabajo-paralelo
spec:
  completions: 3
  parallelism: 2
  template:
    spec:
      containers:
      - name: trabajador
        image: busybox
        command: ["sleep", "10"] # Simula un proceso que tarda 10 segundos
      restartPolicy: OnFailure
```
    Cuándo usarlo: Procesamiento por lotes (batch processing), como procesar una cola de imágenes o correos donde varias tareas independientes se pueden hacer a la vez.

⏱️ Ejemplos de CronJob

Los CronJobs envuelven la definición de un Job dentro de una plantilla (jobTemplate) y añaden la programación (schedule).
1. CronJob que se ejecuta cada minuto

Este CronJob imprime la fecha actual en la consola exactamente cada minuto (* * * * *).
YAML
```
apiVersion: batch/v1
kind: CronJob
metadata:
  name: monitor-minuto
spec:
  schedule: "* * * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: ping
            image: busybox
            command: ["/bin/sh", "-c", "date; echo 'Monitoreo de sistema OK'"]
          restartPolicy: OnFailure
```
    Cuándo usarlo: Envío de métricas periódicas, comprobaciones de estado (health checks) de servicios externos o tareas de limpieza de caché frecuentes.

2. CronJob de "Backup" nocturno con limpieza de historial

Si tienes una tarea diaria (como a la medianoche: 0 0 * * *), no querrás que Kubernetes guarde cientos de Pods terminados en tu clúster. Puedes limitar cuántos registros de Jobs exitosos y fallidos conservar.
YAML
```
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-diario
spec:
  schedule: "0 0 * * *"
  successfulJobsHistoryLimit: 3 # Guarda solo los últimos 3 backups exitosos
  failedJobsHistoryLimit: 1     # Guarda solo el último error para investigar
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: alpine
            command: ["echo", "Simulando copia de seguridad de la base de datos..."]
          restartPolicy: Never
```
    Cuándo usarlo: Copias de seguridad de bases de datos, generación de reportes financieros al final del día, o envíos de correos masivos programados.

📝 Cómo ejecutarlos

Para probar cualquiera de estos ejemplos, simplemente guarda el bloque de código en un archivo llamado tarea.yaml y aplícalo en tu clúster usando kubectl:
Bash
```
kubectl apply -f tarea.yaml
```
Puedes verificar el estado de tus Jobs y CronJobs con:
Bash
```
kubectl get jobs
kubectl get cronjobs
```
